## 摘要
最近的文本聚类方法使用大型语言模型从语料库中提出一个聚类分类体系，然后将每个文本分配到其中。这些流程本质上是程序化的：LLM调用的顺序以及停止、合并和拆分聚类的规则在代码中预先固定，因此它们在不同结构的语料库上泛化能力差，并且难以轻松融入用户提供的约束条件，如目标聚类数量或聚类意图。我们提出了一种代理式替代方案，其中一个编排器LLM在每一步检查发现过程的状态，并分派一小部分专门代理中的一员——提案者、综合者、审计者、调查者和批评者——使流程适应语料库，而不是执行固定流程。在七个公开文本聚类基准上，该方法实现了最先进的性能，在ARI上比先前最强的LLM基线高出最多32%。

## Abstract
Recent text-clustering methods use large language models to propose a cluster taxonomy from a corpus and then assign each text to it. These pipelines are fundamentally programmatic: the sequence of LLM calls and the rules for stopping, merging, and splitting clusters are fixed in code in advance, so they generalise poorly across corpora of different structure and cannot easily incorporate user-supplied constraints such as a target cluster count or a clustering intent. We propose an agentic alternative in which an orchestrator LLM inspects the state of the discovery process at each step and dispatches one of a small set of specialised agents - proposer, synthesizer, auditor, investigator, and critic - adapting the pipeline to the corpus rather than executing a fixed one. On seven public text-clustering benchmarks the method achieves state-of-the-art performance, beating the strongest prior LLM baseline by up to 32% in ARI.

## 方法论

### 核心思想
- 用一个编排器LLM代替固定流水线。编排器在每一步检查当前状态（workspace state），然后分派一个专业智能体。
- 加入用户约束：目标簇数范围 `[k_min, k_max]` 和可选的任务描述（例如“按方法论聚类”）。

### 智能体组成及流程
| 智能体 | 职责 | 触发条件 |
|--------|------|----------|
| **提议者 (Proposer)** | 从语料的不同样本生成候选聚类假设（簇名+描述+证据文本ID） | 初始阶段，多个并行启动 |
| **合成者 (Synthesizer)** | 合并所有提议者的输出，形成单一分类体系，确保簇数在用户指定范围内 | 所有提议者完成后 |
| **审计者 (Auditor)** | 用未见过的新文本测试分类体系，计算覆盖率和置信度，标记弱簇 | 合成后 |
| **调查者 (Investigator)** | 针对审计者或批评者提出的具体问题（如合并/拆分/添加/删除簇），通过检索核实并给出建议 | 审计发现问题时 |
| **批评者 (Critic)** | 全维度评估分类体系（重叠、空白、粒度、边界清晰度、描述质量、簇数是否合规） | 审计通过后 |
- **编排器**：决定何时调用哪个智能体、如何处理返回值（如是否执行调查者的建议、是否触发重新审计）。
- **分类 (Classification)**：最终分类体系用于对全体语料做一次LLM分类（使用GPT-5-mini）。

### 关键技术细节
- 提议者从不同风格（平衡/细粒度/宽泛）出发采样，样本大小按文本长度动态调整。
- 合成者通过比较簇名称描述和检查证据原文来判断合并，保留仅出现在一个提议者中的簇但标记低置信度。
- 审计者从语料中抽取全新样本逐文本验证，输出置信度分数（1-5），低于阈值的簇被标记。
- 调查者使用TF-IDF检索、簇分配文本或特定文本ID来获取证据，形成六种建议（合并、拆分、重命名、添加、删除、不变）。
- 批评者沿6个维度评估，发现k范围违规时触发重新合成。

## 实验设计

### 数据集与场景
- **7个公开基准**：
  - **短文本**：Banking77（银行查询，77类）、CLINC150（助理查询，150类+无关类）、MASSIVE-Intent（60类）、MASSIVE-Domain（18类）、GoEmotions（Reddit评论，27种情绪+中性）、StackExchange（论坛标题，121类）
  - **长文本**：20 Newsgroups（Usenet帖子，20类，平均1200字符）
- 覆盖不同数据类型（短查询、长文档、多簇、含噪声类）和聚类粒度。

### 对比方法
- **传统概率模型**：LDA
- **稠密嵌入+经典聚类**：SBERT+k-means、BERTopic
- **LLM增强嵌入**：LLM-embedding+k-means（OpenAI text-embedding-3-large）、ClusterLLM
- **LLM驱动分类体系生成**：TopicGPT、Huang & He
- 所有LLM方法统一使用Claude Opus 4.7（粗粒度任务）和GPT-5-mini（细粒度分类）。

### 评估指标
- Adjusted Rand Index (ARI)、Normalized Mutual Information (NMI)、聚类准确率ACC (Hungarian assignment)。
- 分**给定k**（使用真实簇数）和**发现k**（方法自动决定簇数）两种设置。

## 主要结论与发现

1. **Agentic Clustering在所有基准上达到SOTA或持平**，在大多数数据集上ARI比第二好方法高出超过10%。
2. **其他LLM分类体系方法（TopicGPT、Huang & He）粒度控制困难**：没有用户意图提示时，要么产生太多小簇，要么太少大簇，性能甚至不如LDA。
3. **消融实验表明**：
   - 去掉**任务描述**显著降低性能（尤其在MASSIVE-Domain、CLINC150上），说明聚类意图的重要性。
   - 去掉**后处理循环（审计者、批评者、调查者）**影响很小，表明主要性能提升来自提议者采样、提示和合成阶段。
4. **成本效益**：虽然略高于嵌入方法，但质量领先，且并非最贵方法（Huang & He更高）。

## 优点

1. **灵活性**：编排器动态调度，自适应语料结构，而非固定流水线。
2. **可控性**：接受用户指定的簇数范围和聚类意图，使聚类结果符合用户需求。
3. **多智能体分工**：专门化代理（提议、合成、审计、调查、批评）各司其职，提升鲁棒性。
