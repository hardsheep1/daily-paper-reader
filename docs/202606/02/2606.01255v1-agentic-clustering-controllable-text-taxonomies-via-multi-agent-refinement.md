## 摘要
最近的文本聚类方法使用大型语言模型从语料库中提出聚类分类体系，然后将每个文本分配到其中。这些流程本质上是程序化的：大型语言模型调用的序列以及停止、合并和拆分聚类的规则事先在代码中固定，因此它们在不同结构的语料库上泛化能力差，并且难以轻松集成用户提供的约束条件，如目标聚类数量或聚类意图。我们提出了一种智能体替代方案，其中编排型大型语言模型在每一步检查发现过程的状态，并分派一组专门智能体（提议者、合成者、审计者、调查者和批评者）中的某一个，使流程适应语料库，而不是执行固定的流程。在七个公开的文本聚类基准测试中，该方法实现了最先进的性能，在ARI上击败了之前最强的大型语言模型基线高达32%。

## Abstract
Recent text-clustering methods use large language models to propose a cluster taxonomy from a corpus and then assign each text to it. These pipelines are fundamentally programmatic: the sequence of LLM calls and the rules for stopping, merging, and splitting clusters are fixed in code in advance, so they generalise poorly across corpora of different structure and cannot easily incorporate user-supplied constraints such as a target cluster count or a clustering intent. We propose an agentic alternative in which an orchestrator LLM inspects the state of the discovery process at each step and dispatches one of a small set of specialised agents - proposer, synthesizer, auditor, investigator, and critic - adapting the pipeline to the corpus rather than executing a fixed one. On seven public text-clustering benchmarks the method achieves state-of-the-art performance, beating the strongest prior LLM baseline by up to 32% in ARI.

# 1. 核心问题与整体含义（研究动机与背景）

- **问题背景**：大型语言模型（LLM）被越来越多地用于文本聚类，核心思路是让LLM从语料库中提出一个聚类分类体系（taxonomy），然后将每个文本分配到该体系中的某一类。现有方法采用**固定的程序化流水线**（programmatic pipeline）：LLM调用的顺序、停止/合并/拆分规则、每步可以查阅的证据类型均在代码中预先硬编码。
- **核心缺陷**：
  - **泛化性差**：无法适应不同结构的语料库（如短文本 vs 长文档、不同粒度）。
  - **缺乏用户可控性**：难以融入用户约束，如目标簇数范围（k-range）或聚类意图（intent，如“按方法论聚类” vs “按主题聚类”）。
  - **无法自适应**：面对不同粒度或噪声比例的数据集时，性能可能不如基于稠密嵌入的传统方法。
- **研究动机**：聚类问题没有唯一正确答案，应根据语料动态调整。因此，本文提出**智能体式聚类（Agentic Clustering）**，用编排型LLM动态决定下一步调用哪个专门智能体，代替固定流水线。

# 2. 方法论：核心思想、关键技术细节与流程

- **核心思想**：不再执行固定的LLM调用序列，而是由一个**编排器（orchestrator）** 在每一步检查当前状态，并分派一组**专门智能体**中的某一个。这些智能体各自负责单一任务，且共享用户提供的约束（k-range和聚类意图）。整个流水线是动态组装的，而非预先硬编码。

- **关键技术细节**：
  - **输入**：未标注语料库 D、目标簇数范围 [k_min, k_max]、可选的自由文本聚类意图描述（如“按问题类型聚类”）。
  - **智能体组成**：
    1. **提议者（Proposer）**：从语料中抽取不相交样本，生成候选簇集（每个簇包含名称、自然语言描述、证据文本ID）。多个提议者并行运行，采用不同风格（平衡、细粒度、粗粒度），以便下游合成者识别稳健结构。样本大小根据文本长度动态调整。
    2. **合成者（Synthesizer）**：读取所有提议者的输出，合并重复概念，形成唯一的建议分类体系。合并依据包括自然语言重叠和直接检查证据文本。输出结果必须落在指定k范围内，单提议者支持的簇被保留但标记为低置信度。描述遵循质量规则（定义概念，而非总结样本）。
    3. **审计者（Auditor）**：用从未看过的新样本测试建议分类体系。对每个文本，分配最佳匹配簇并给出置信度（1-5），无法匹配则标记为“none”。输出覆盖率和弱簇列表。
    4. **调查者（Investigator）**：当审计者或批评者发现问题时，针对具体问题（如“是否应合并c3和c5？”）发出检索查询（TF-IDF、基于聚类分配、特定ID），读取检索到的文本，形成六种建议之一：合并/拆分/重命名/添加/删除/无变更。每个建议附带证据和推理。
    5. **批评者（Critic）**：在审计通过后评估分类体系的六个维度：重叠、缺口、粒度、边界可区分性、描述质量、k范围合规性。每个维度给出通过/警告/失败。驱动编排器下一步动作（如触发调查、改写描述、重走合成）。
  - **编排器（Orchestrator）**：唯一的持久组件，不直接读语料，也不修改分类体系。它根据状态决定下一个调用的智能体、处理返回值、决定何时终止（当批评者六个维度全部通过时）。
  - **最终分类**：用最终分类体系对全语料进行LLM分类（单个文本独立调用LLM，输出簇ID或“none”）。可选地，可针对少量人工标注子集优化分类提示。

- **流程图示**（论文Figure 1）：
  1. 语料样本 → 并行多个提议者 → 各自输出提案。
  2. 合成者合并 → 建议分类体系。
  3. 审计者测试新样本 → 若有问题，调查者介入 → 修改后重新审计。
  4. 批评者评估 → 若通过，则最终分类体系 → LLM分类器 → 全语料分类。

# 3. 实验设计

- **数据集（7个基准）**：
  - Banking77（银行查询，77个意图，3080条）
  - CLINC150（助手查询，150个意图+OOS，5500条）
  - MASSIVE-Intent（助手查询，60个意图，2974条）
  - MASSIVE-Domain（同一批语料，18个领域）
  - GoEmotions（Reddit评论，27种情绪+中性，45446条）
  - 20 Newsgroups（Usenet帖子，20个主题，18331条）
  - StackExchange（论坛标题，121个站点，4156条）
- **对比方法**：
  - 传统方法：LDA（概率主题模型）
  - 稠密嵌入方法：SBERT + k-means、BERTopic、LLM-embedding + k-means（OpenAI text-embedding-3-large）、ClusterLLM

- **指标**：调整兰德指数（ARI）、归一化互信息（NMI）、聚类准确率（ACC，匈牙利分配）。
- **设置**：
  - 所有LLM方法统一使用Claude Opus 4.7用于重任务（分类体系生成/合并/精炼），GPT-5-mini用于轻任务（全语料分类）。
  - 给k的方法使用真实k，发现k的方法（BERTopic、TopicGPT、Huang & He、本文方法）报告两种配置。
